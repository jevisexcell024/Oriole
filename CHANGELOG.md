## v1.12.1 — Custom-role privilege-escalation fix

**Found during a live authorized penetration test.** `POST /api/admin/roles` (create a custom role, gated on `roles.manage`) accepted any set of permissions with no check against what the caller themselves actually holds, and `PATCH /api/admin/team/:id/custom-role` (assign a custom role, gated on `roles.team_manage`) let the caller assign any existing role to any staff member — including their own account, with no self-assignment guard at all. Together, a staff member holding only those two keys (a realistic "delegate role management" grant, not full admin) could mint a role containing every permission in the system — `org.manage`, `system.settings`, `students.delete`, everything — and assign it to themselves, reaching full admin-equivalent access without ever holding an admin permission directly. Two API calls, no admin credentials required.

**Fix**: a new `permissionOverreach()` ceiling check (`server/auth.ts`) — a role-manager can now only grant or assign permissions they themselves currently hold, whether directly or through inheritance. Applied to all three custom-role write endpoints: create, update, and assign. Verified against the live server: reproduced the full exploit chain end-to-end (limited "role delegate" account → attempt to mint an all-permissions role → attempt to self-assign a pre-existing powerful role), both steps now return `403`. Four new pinning tests in `test/permissions.test.ts` cover the fix directly so this can't silently regress.

**Also**: the breached-password check (`server/schemas.ts`, Have I Been Pwned range API) fails open by design on any network error — a deliberate, documented tradeoff so a slow/down third party never blocks account creation. Added a warning log on that path, since a host with no outbound internet access (common on locked-down shared hosting) means the check silently never runs for *any* password, with nothing in the logs to reveal it. Worth confirming the production host can actually reach `api.pwnedpasswords.com`; if not, only the baseline length/variety rule (12+ characters, 5+ distinct characters) is actually protecting new passwords today.

## v1.12.0 — Codebase Cleanup, Database Optimization & Security Review (Enterprise UX, Workstream 3)

**Third workstream of the Enterprise UX release** (Workstreams 1 — navigation — and 2 — UI/dashboards — shipped separately). Covers codebase cleanup, database optimization, and a security review — deliberately excluding the API response-envelope standardization, which stays deferred to its own isolated, high-risk pass. Three parallel audits found this codebase in good shape already; the following are the genuine, concrete issues found, not a speculative rewrite.

**Codebase cleanup.** The audit turned up remarkably little — no stray `console.log`s, no commented-out code, no TODO/FIXME markers, no dead files, no unused dependencies. The one real finding: `initials(name)` and a relative-time `fmtAgo` formatter were byte-for-byte duplicated across several pages. Both are now shared helpers in a new `src/lib/format.ts` (`initials`, `timeAgo`), consolidating `AdminCandidates.tsx`, `AdminClasses.tsx`, `AdminTeam.tsx`, `AdminDashboard.tsx`, `StudentsSIS.tsx`, and `AdminResults.tsx`. Four *other* "time ago" variants elsewhere in the app have real behavioral differences (rounding, i18n-key usage, fallback thresholds) and were deliberately left alone rather than forced into one function. Also moved 4 `@types/*` packages from `dependencies` to `devDependencies` in `package.json` (compile-time only, no runtime effect).

**Database optimization.** This app's storage is a document store (in-memory JS arrays mirrored to generic `(id, doc jsonb)` Postgres/PGlite tables), not a relational schema — so "optimization" here means algorithmic complexity, not SQL indexes (the 9 high-volume off-mirror tables — snapshots, proctor events, audit logs, etc. — already have correctly-targeted SQL indexes). `GET /api/admin/results` did repeated linear scans over `users` and `exams` inside a loop over every attempt, and a linear scan over every attempt inside a loop over every exam — now replaced with `Map`s built once per request, finishing a pattern the codebase already uses elsewhere (`server/index.ts`'s violations endpoint). Also closed a real data-integrity gap: deleting a candidate cascaded their attempts/registrations/certificates/answers but left their id sitting in any class's `memberIds` array — now cleaned up too. Full server-side pagination and a broader `classes`-deletion cascade were identified but are larger, frontend-touching changes out of scope for this pass.

**Security review.** Overall posture is good, and most of what a fresh audit would flag turned out to be a *deliberate, already-documented* tradeoff rather than a gap: bcrypt cost 8 (tuned against a real login-storm load test), no CSRF token (relies on `SameSite=Lax` cookies + same-origin deployment — there's no CORS configured, so no cross-origin credentialed request is even possible), CSP's `unsafe-eval`/`unsafe-inline` (required by the Monaco code editor and React inline styles), and fail-open malware/breach-check scanning (so a third-party outage never blocks an exam submission). None of these were touched — changing any of them risks breaking Monaco, login-storm resilience, or account creation without the full context those tradeoffs were made with; they're recorded here as a reviewed decision rather than an implicit risk. The one genuine, fixable gap: `src/pages/Library.tsx` rendered `mammoth`-converted DOCX HTML via `dangerouslySetInnerHTML` with no sanitization step — the only unsanitized HTML-injection point found anywhere in the codebase. Added `dompurify` and sanitize the HTML before it's ever set as state; verified directly against the bundled version that it strips `<script>`, inline event handlers, and `javascript:` URLs while preserving headings, formatting, lists, tables, and images untouched.

## v1.12.0 — UI/UX Standardization, Dashboards & Settings Cleanup (Enterprise UX, Workstream 2)

**Second workstream of the Enterprise UX release** (Workstream 1 — navigation/breadcrumbs/sidebar/icons — shipped separately). Covers UI/UX standardization, dashboard cleanup, component refactoring, and settings cleanup. A three-pass audit of all 74 `src/pages`/`src/components` files found the real, concrete inconsistencies (not a guess-and-rewrite) — this release fixes the ones with a clear, low-risk payoff and leaves broader retrofits (badges, full typography pass) for a future incremental pass.

**Facilitators and proctors get real, focused dashboards.** Previously they saw the exact same admin overview — org-wide Institution Health, Academic Insights, and charts they had no capability to act on. `AdminDashboard.tsx` now gates every card behind the same `can(role, cap)` system that already filters the sidebar nav: a proctor (`monitor` cap only) sees Live Monitoring and AI Proctoring; a facilitator (`grading`/`results`/`monitor`/`communication`) additionally sees Quick Actions, Exam Activity, and their grading queue — never the org-wide stats they can't access anyway. Admin is unaffected.

**Dashboard decluttered.** Fixed the confirmed internal duplication found during the audit: "Create Exam" no longer appears both in the header and in Quick Actions; the pending-review count no longer appears under three different labels (Pending Reviews stat, a mislabeled "In Review" AI-proctoring stat, and Needs Attention); "Flagged" no longer appears in both Live Monitoring and AI Proctoring; and the "Completion Trends" chart — which plotted the exact same dataset as the Exam Activity chart above it, just with a different series pair — was removed.

**New shared components replace three copy-pasted ones.** `<Modal>` and `<ErrorBanner>` (`src/components/ui.tsx`) replace a byte-for-byte-identical local `Modal` implementation duplicated across `AdminCandidates.tsx`, `AdminClasses.tsx`, and `AdminTeam.tsx`, and the same red error-banner markup that had been copy-pasted across 27 files. A new `.btn-danger` class in `index.css` codifies the solid-red destructive-button treatment 5 call sites were already hand-rolling.

**Settings cleanup.** `AdminSettings.tsx`'s local `Settings` interface declared `name`/`supportEmail`/`website`/`timezone` fields that were never actually rendered — those org-profile fields live on `AdminOrganization.tsx`, which edits the same backing record. Removed the dead fields; left Organization/Settings/Integrations as separate pages rather than merging them, since they're each a reasonable size and merging would retrain navigation muscle-memory for no functional gain.

**Typography — targeted, not a full retrofit.** Added `.h1`/`.h2`/`.h3` size-scale utility classes to `index.css` for future use, and removed a genuinely redundant pattern found in `AdminDashboard.tsx` and `AdminExams.tsx`: both files locally re-applied a `fontFamily: DISPLAY` inline style to `<h1>`/`<h2>`/`<h3>` elements that already inherit the same font from a global `@layer base` rule. Full page-by-page heading-size standardization across all 55 pages is out of scope for this pass.

## v1.11.0 — Media Comprehension questions (Language & Media Assessment, Phase 2)

**A new question type, not five.** `media_comprehension` covers Audio, Video, Image, PDF/Document and Reading-Passage comprehension in one flexible type: pick a stimulus kind, then pick an answer format (MCQ, Multi-select, Short Answer, Numeric, Essay, Matching or Cloze) — the question reuses that format's existing editor, delivery, and grading exactly as it already works for its native type, rather than duplicating each combination. Reading passages support "one passage, several independently-scored questions" via a shared passage group, so a comprehension passage can carry a short series of questions without repeating the text.

**Player & security posture.** Audio/video get play/pause, a replay-limit with a visible remaining-plays indicator, an optional countdown before playback, an optional playback-speed control, and best-effort seek-blocking and download-prevention (`controlsList="nodownload"`, no `download` attribute, right-click disabled) — the same "deterrence, not a guarantee" posture already documented for exam lockdown, not a new security model. Images support zoom/pan/fullscreen and side-by-side comparison for multiple exhibits. PDFs render through the browser's own native viewer (zoom/page-nav/search come free) rather than a custom-built one — a deliberate scope reduction, and the browser's own PDF toolbar may still offer a download control that can't be fully suppressed short of writing a PDF.js-based renderer.

**Storage — lean, not new infrastructure.** This app has no object storage, no signed URLs, and no HTTP Range streaming anywhere — every existing "media" field (webcam snapshots, ID photos, `file_upload` answers) is a size-capped base64 blob in a Postgres/PGlite `jsonb` column. Media assets extend that same pattern into their own off-mirror table (so multi-MB blobs never sit in the in-memory `questions` table) with size caps between the existing 2.5MB `file_upload` cap and the 25MB digital-library cap (image 5MB, audio 15MB, video 40MB, PDF 20MB), the same malware-scan/markup-sniff validation `file_upload` already uses, and an `externalUrl` escape hatch for anything longer, mirroring the digital library's own `Book.externalUrl`. Media is never embedded inline in the exam payload or exposed at a public URL — it's served through an authenticated, ownership-checked endpoint (a candidate can only fetch an asset that's actually attached to a question in their own attempt); this is real access control, not full CDN-grade signed-URL streaming, and is called out here rather than overclaimed.

**Grading & reporting reuse.** `gradeOne()` delegates a media question's grading to whatever its answer format already does (auto-graded for MCQ/Numeric/etc., manual review for Essay) in one line — no new grading logic. The event timeline, the manual-grading rubric panel, and the correct/candidate-answer display all already render media questions correctly with no per-type special-casing beyond what every prior question type needed. A new **Media Usage** card (plays, replays, minutes) on the Integrity page mirrors the existing Calculator Usage card exactly, explicitly labeled as usage reporting, not a violation indicator — playing exam media is never auto-flagged.

**Modular by design.** Off by default, org-wide — `Settings → Media & Language Assessment` — so an institution opts in explicitly; already-authored media questions keep working if the toggle is later turned off, only the Exam Builder's type picker hides.

**Deferred to Phase 3** (student audio/video recording for speaking assessments — the second half of the original feature request) as a separate, focused follow-up once this phase is verified in use, along with AI pronunciation scoring/transcription (explicitly out of scope for now, per the original request) and DOCX/Slides document conversion.

## v1.10.0 — Exam calculator & screen wake lock

**Built-in exam calculator.** A new per-exam, admin-configurable calculator (`Exam Builder → Proctoring Rules → Calculator`) that renders entirely inside the exam UI — never the OS calculator, never a new window. Basic (+, −, ×, ÷, %, √, ±) and Scientific (trig, log/ln, powers, factorial, π/e, DEG/RAD toggle) modes, a draggable floating window or fixed sidebar, optional keyboard input, optional in-calculator history (client-only, never sent to the server), and an optional "remember state between questions" setting (on by default) so the display survives question navigation. Copy/paste, right-click and text selection are blocked on the display. All arithmetic runs in a new pure reducer (`shared/calculator.ts`, 25 pinning tests) shared between the component and its tests.

**Screen Wake Lock.** A new "Power Management" section in the same Proctoring Rules tab requests the browser's Screen Wake Lock API for the duration of an exam, re-acquiring it automatically after tab-visibility changes. If the browser doesn't support the API, a non-blocking banner (or nothing, or a proctor notification — admin-configurable) tells the candidate without ever stopping the exam. Extended device-inactivity is detected via a visibility-change/heartbeat check and reported to a new server endpoint, which — per the exam's "On Device Resume" setting — can simply log it, pause the attempt for proctor review (same as a manual pause), or auto-submit.

**Reporting, not accusation.** Every calculator open/close/minimize and wake-lock event is logged through the existing proctoring pipeline at `info` severity (never counted toward integrity deductions or the violations list) unless a wake-lock resume gap is ≥30s, matching the spec's own "should not automatically flag cheating." A new **Calculator Usage** card on the Integrity page shows per-student open counts and total time, explicitly labeled as usage reporting only. Fixed a related visibility gap while building this: the per-attempt event timeline on the attempt review page was gated to proctored exams only, which would have hidden these events on a non-proctored exam that still had the calculator or wake lock enabled — it now shows whenever there's anything to show.

## v1.9.0 — Status & Reliability Center

A new **Status & Reliability Center** (`/admin/reliability`), plus a simplified public status page (`/status`) — real-time health monitoring, automatic incident detection, and a feature unique to an exam platform: **Exam Impact Analysis**, which tells an institution exactly which exams and students an incident affected and whether integrity was maintained.

**Honest by design.** Oriole runs as a single process (no separate microservices, no job queue) — rather than faking independent "services," this monitors **7 real, independently-measurable subsystems** (API, database, exam delivery, auth, notifications, file storage, background jobs), each backed by genuine per-route latency/error instrumentation, a real `db.ping()` round-trip, real `process.memoryUsage()`/`os.loadavg()`, and the existing mailer/SMS status checks — every number shown is a real measurement, never simulated (matching the standard set by the earlier "Truth & trust fixes" release).

**Automatic incident lifecycle.** A 60-second health-check sweep classifies each subsystem as operational/degraded/down; two consecutive bad ticks open an incident (debounced against a single blip), which immediately computes its exam impact, notifies configured channels, and auto-resolves after three consecutive clean ticks — the full timeline (opened → identified → notified → monitoring → resolved) is recorded per incident. Admins can also resolve an incident manually or trigger an on-demand health check.

**Exam Impact Analysis.** Every incident computes, from real attempt data: how many exams and students were active during the incident window, how many attempts auto-recovered cleanly, how many need manual review, and how many were actually lost — using the platform's own existing overdue-attempt logic, not a separate approximation. The integrity verdict ("maintained" or "review recommended") is always backed by a specific, inspectable basis — a lost-attempt count or a real database-restore event found in the tamper-evident audit log — never asserted blindly.

**Uptime & performance history.** 24h/7d/30d/90d/365d uptime timelines and 1h/24h/7d/30d response-time graphs (avg/min/max/p95/p99), backed by a nightly rollup job that aggregates raw samples (kept 7 days) into permanent daily summaries — keeping the 365-day view cheap without unbounded raw-sample storage. A computed, explainable **reliability score** (uptime + latency + recovery speed + exam impact, weighted) with an on-demand AI plain-English summary (reuses the existing difficulty-checker/trend-narrative AI provider, hidden when not configured).

**Notifications** — incident open/resolve events dispatch through the existing SSRF-guarded webhook system (Slack/Teams/Discord all receive a signed JSON payload today) plus email and SMS to configured recipients (new Settings → Reliability Alerts section); a "degraded" reading only notifies if explicitly opted in, "down" always notifies.

**Public status page** (`/status`, no login) shows exactly what's appropriate for an external audience — overall status, uptime percentage, per-service status, incident history — and nothing else: no latency numbers, no reliability score, no exam/student data, served from dedicated public endpoints (not the admin ones with a flag) so a future admin-side change can't accidentally leak internal data onto it.

**Deferred to a later release** (no such tooling exists in this codebase today): native Slack/Teams/Discord rich-embed formatting, PDF/Excel export (CSV ships now via the existing report pipeline), and WebSocket/SSE push (the dashboard polls, matching Live Monitor's existing pattern). New `test/reliability*.test.ts` (5 files) cover the status-classification math, storage layer, sweep mechanics, incident/exam-impact bucketing, rollup/purge, and public-data redaction.

## v1.8.0 — Custom roles & permissions, forced first-login password change, reliability fixes

**Custom-roles / permissions system.** Access control was previously three hardcoded roles (admin/facilitator/proctor). A new additive layer lets an institution define its own roles: a granular permission catalog (~31 keys spanning exams, monitoring, grading, results, students, communication, org structure, system, role management and the library), role **inheritance** from a system role or another custom role, **org-unit scoping** (faculty/department/campus), and **time-limited grants** (auto-expiring role assignments). New **Roles** admin page to create/edit/delete roles and see who holds each one; deleting a role that's still assigned or has children prompts a reassign-and-delete flow instead of blocking or silently orphaning them. All ~120 admin/staff endpoints were migrated onto the new permission checks (the original three roles keep their exact same default capabilities — this is additive, not a behavior change for existing installs).

**Forced password change on first login.** Students now must set their own password the first time they sign in with a temporary one (new account, resent invite, admin-driven reset, or roster import) — a full-screen, non-dismissible step right after login. This is a narrow, one-time exception to the existing "students can't self-service their password" policy (which stays in place for every login after this one) — recommended by the June 2026 security audit (M2) and closes it.

**Reliability**
- **Graceful-shutdown fix** — the shutdown handler's timeout fallback (used when an in-flight request takes too long to finish during a restart) now closes the database before exiting; previously it could exit without doing so, risking the same embedded-database corruption an unclean kill causes. Triggered by a real production incident (an unclean shutdown during a deploy corrupted the embedded store; scheduled auto-backup recovery restored it with a small, bounded loss window).
- **Permissions-Policy header regression fixed** — the SPA-fallback route was overwriting the security header set earlier in the request, silently dropping the `geolocation` grant the v1.7.0 geofencing feature needs (`res.setHeader` replaces rather than merges). Caught in production; fixed so the fallback route no longer sets its own copy of the header.
- **Live camera/location permissions diagnostic** — System Health gained a "Camera/Location Access" check that reads back the *actual* header a browser receives (not just what the code intends to send), with a per-feature allowed/blocked/missing breakdown and a Recheck action — built specifically to make this class of regression diagnosable instead of guessed at, since three independent places can set this header (app middleware, the SPA fallback, and the production host's `.htaccess`).

## v1.7.0 — Geofencing: entry check + continuous monitoring

**Phase 1 — entry check.** Exams can now be restricted to approved physical locations: admins draw one or more approved areas on a map in the exam builder (search-by-address, current-location, or manual coordinates, each with its own radius), and a candidate must be inside one of them — within a configurable GPS-accuracy tolerance — to start the exam. Every check (pass or fail) is logged to a new `geofence_logs` table for a full location audit trail, independent of whether the candidate was let in.

**Phase 2 — continuous monitoring during the exam.** Entry alone doesn't stop someone leaving mid-exam, so geofenced exams can now optionally re-check the candidate's location on a configurable interval for the whole duration:
- **Configurable exit policy** (from most to least lenient) — *warn* (log only, exam continues), *pause* (freeze the timer until they return — the candidate loses no time), *lock* (block the exam screen but the timer keeps running — stricter than pause), or *auto-submit*/*terminate* (end the exam immediately, routed through the same `forceSubmitAttempt` path the existing violation-limit auto-submit already uses).
- **Grace period** — a candidate who steps outside gets a configurable window (with a live countdown shown in the exam UI) to return before the policy actually applies.
- **Server-enforced, not just client-trusted** — a background sweep independently re-applies the exit policy every 30s, so a candidate can't dodge it by simply letting their browser stop sending location pings instead of returning.
- **Live location map** — `Live Monitor` gained a Grid/Map toggle; map view plots every candidate on a geofenced exam's approved area(s) in real time (green = inside, red = outside, grey = no GPS fix), and the existing intervene drawer gained a Location panel plus a "clear geofence lock/pause" override for proctors (e.g. for a GPS glitch — doesn't require the candidate to physically move first).

**Disclosed trust boundary:** browser geolocation is client-reported and inherently spoofable — this is a known, accepted limitation shared with Phase 1's entry check, not something Phase 2 changes. Accordingly the *lock* policy's exam-screen block is UI-only enforcement (like the existing fullscreen-lock overlay); the real teeth is server-side — the independent sweep and the *auto_submit*/*terminate* policies, which cannot be bypassed by a client that stops cooperating.

New `shared/geo.ts` helpers (`graceSecondsRemaining`, `graceExpired`) are pure and unit-tested (`test/geo.test.ts`), matching the existing haversine/`nearestGeofence` coverage.

- **Low** — the announcement read-receipt endpoint didn't verify the announcement was in the caller's audience before recording a read. Fixed.
- **Low** — `Book.externalUrl` accepted any URL scheme. Now restricted to `http(s)://`.
- Added `test/library.test.ts` and `test/uploads.test.ts` — this new logic previously had zero automated coverage.

## v1.5.1 — Reliability, operations & fixes

- **Scheduled database backups** — the embedded database previously had no backup strategy at all (single folder, single host, no recovery path if the disk failed). Now takes a full compressed snapshot of every table on a schedule (default: daily, keeps the last 14) plus a manual "Run backup now" action, written to a folder that survives redeploys (a sibling of the live data directory, not inside the app's own redeployable code). Surfaced as a new status card + detail panel on the System Health page. New `server/backup.ts`, `GET/POST /api/admin/backup/*`.
- **Exam duration changes now propagate live** — increasing an exam's duration while students are mid-exam now updates their live countdown within seconds (previously only affected attempts started after the change).
- **Live countdown to next scheduled exam** — a large, prominent countdown on the student dashboard for an upcoming scheduled exam, turning urgent (and pulsing) inside the final hour.
- **Exam delete** — admins can now delete an exam from the Exam Library (previously add/edit-only, no way to remove one).
- **Word (.docx) question-bank import fixed** — column headers like "Question" / "Correct Answer" weren't being recognized (the importer only matched exact internal field names); it now falls back across common header variants.
- **Mailer completion** — result-released, registration-confirmed, exam-reminder, and admin-digest emails now all send full branded HTML (previously plain text only, or only partially built out). SMTP connectivity is verified at server startup so a misconfiguration surfaces immediately in the logs instead of silently failing on the first real send.
- **Fix: unnecessary full-database rewrites on live-exam admin actions** — the exam-settings save endpoint was rewriting every row of every table on each save (a pattern meant for rare bulk operations, not routine edits). Now only writes the specific rows that actually changed, avoiding a potential stall on concurrent student traffic once an org has accumulated significant historical data.

## v1.5.0 — Security hardening

A dedicated security pass across two work sessions: an initial audit, then a follow-up adversarial review that dug further into auth, data-at-rest, dependencies, and infrastructure. Overall verdict going in: the codebase was already fairly security-conscious (parameterized SQL throughout, field-whitelisted PATCH handlers with no mass-assignment risk, `timingSafeEqual` SEB verification, bcrypt, candidate-scoped queries) — this release closes the specific gaps found and adds automated checks so regressions get caught going forward.

**Authentication & session security**
- Hardcoded admin password removed — set via `ADMIN_PASSWORD`, else a random one-time password is generated and printed once at first boot.
- Hardcoded JWT dev-default removed — an unset `JWT_SECRET` now gets a random per-process key (sessions reset on restart, never forgeable with a known default); `assertProductionEnv()` still requires a real one in production.
- Session revocation via a per-user `tokenVersion` — logout, password change, and admin-initiated resets all bump it, immediately invalidating any existing session token.
- Password policy strengthened to a 12-character minimum with a variety check (was 6).
- 2FA (TOTP) secrets are now encrypted at rest with the same AES-256-GCM field encryption already used for avatars/snapshots — previously stored in plaintext, meaning a database leak would have handed over every account's second factor.
- 2FA replay protection — a captured or observed authenticator code can no longer be reused a second time within its ~30s validity window.
- 2FA brute-force limiter added to the verify endpoint.
- Login timing side-channel fixed — a login attempt for a nonexistent email previously returned near-instantly while a wrong password on a real account took ~100ms (bcrypt), letting an attacker enumerate registered emails by measuring response time. Both cases now take the same time.
- SSO (Microsoft) `id_token` now validates audience, issuer, and expiry, not just that it arrived over the token exchange.

**Data protection**
- Check-in selfie, photo-ID, and room-scan photos encrypted at rest, with a boot-time migration for any existing plaintext rows.
- Webhook SSRF guard — outbound webhook URLs must be `https://` and are rejected if they resolve to a private/loopback/link-local/cloud-metadata address; re-validated at every delivery (not just at creation), closing a DNS-rebinding window.
- Audit log made tamper-evident — every entry hash-chains to the one before it, so any later edit or deletion breaks verification (surfaced on the Audit Logs page).
- File-upload MIME allowlist plus content sniffing — rejects markup/script payloads even when the declared content type is spoofed.
- CSV export formula-injection fix — a candidate's self-editable display name could otherwise execute as a live formula (e.g. `=HYPERLINK(...)`) the moment an admin opened a results/students/certificates export in Excel or Sheets.
- Outbound email HTML injection fix — user- and admin-controlled values (name, exam title, organization name) are now escaped before being interpolated into email HTML.
- Proctor-event submissions validated against a known event-type allowlist, severity clamped to a fixed set, and message length capped (previously candidate-controlled and unbounded).
- Embedded database (`.pgdata`) file permissions locked down on the live host — found world-readable during the audit, restricted to owner-only.

**Dependencies & CI**
- `npm audit`: 7 vulnerabilities → 0. Upgraded nodemailer 6→9 (high — SMTP/header injection), swapped `xlsx` to SheetJS's official CDN build, and forced the transitive `dompurify` pulled in by `monaco-editor` to a patched version via a `package.json` override (monaco-editor pins an exact vulnerable version, so normal dependency resolution couldn't fix it on its own).
- CI security gates added to `.github/workflows/ci.yml`: `npm audit --audit-level=critical`, CodeQL (security-extended), and gitleaks secret-scanning.

**Reviewed and confirmed solid (no change needed):** IDOR — every attempt/registration/certificate endpoint scoped by candidate ID across all ~150 routes; no state-mutating GET endpoints (no SameSite=Lax CSRF bypass surface); no CORS misconfiguration; request logs never capture request bodies (login/signup passwords are never logged); IP addresses used only as in-memory rate-limit keys, never persisted or exposed in any response.

**Known, accepted tradeoffs:** CSP allows `unsafe-eval`/`unsafe-inline` in `scriptSrc` (required by the Monaco code editor); not multi-tenant (single shared namespace — safe run as one instance per institution, not as shared SaaS); web-based proctoring is inherently bypassable by a sufficiently motivated candidate on their own hardware — Safe Exam Browser is the only hard, OS-level lock; this is a fundamental limit of browser-based invigilation, not a bug in this codebase.

## v1.4.0 — New features

Net-new capabilities (built on the existing Express + embedded-PGlite backend — no data migration required).

**Shipped**
- **SSO — Sign in with Microsoft (Entra / Microsoft 365)** — OAuth authorization-code flow, **added alongside** the existing email + password login (the button only appears when configured). It signs in **only accounts that already exist in Orcalis** (matched by email) — SSO never creates accounts, so a tenant member can't self-provision. CSRF-protected via a state cookie; 2FA is delegated to the identity provider. Configure with `MS_CLIENT_ID` / `MS_CLIENT_SECRET` / `MS_TENANT` (and register the `…/api/auth/sso/microsoft/callback` redirect URI in Entra). New `server/sso.ts`, `/api/auth/sso[/microsoft/start|/microsoft/callback]`. **This completes the v1.4.0 backlog** (i18n continues to expand screen-by-screen).
- **Multi-language UI (i18n) — foundation + first screens** — a lightweight in-house i18n engine (`src/lib/i18n.tsx` + `i18n-messages.ts`, no new dependency) with a **language switcher** (`LanguageSwitcher`) and five languages: **English, French, Spanish, Portuguese, Arabic** (Arabic switches the document to **RTL**). Language persists in localStorage. The **login** screen and the **student portal navigation/greeting** are fully translated; remaining screens use the same `t("key")` pattern and migrate incrementally (untranslated text falls back to English). The catalogue's typing forces every key to include all five languages.
- **Webhooks + public API** — a new admin **Integrations** page (`/admin/integrations`). **Webhooks:** subscribe a URL to events (`attempt.submitted`, `result.released`, `certificate.issued`, `exam.published`); Orcalis POSTs a JSON payload signed with HMAC-SHA256 in an `x-orcalis-signature` header, with per-webhook secret, active toggle, last-delivery status, and a test ping. **Public read-only API:** generate API keys (shown once, stored sha256-hashed) and call `GET /api/v1/{exams,results,certificates}` with `Authorization: Bearer <key>`. New `OrgSettings.webhooks/apiKeys`, `dispatchWebhook()` wired into the four event sites, `requireApiKey` middleware.
- **Report builder + scheduled exports** — the Reports page now takes an optional **date range** (filters the Results and Certificate exports by submission/issue date) and supports **scheduled exports**: pick a report + daily/weekly cadence + recipient emails, and the server emails the CSV automatically each period (covering the trailing day/week) via a new hourly sweep. New `OrgSettings.scheduledReports`, a shared `buildReportCsv()` (reused by downloads + the sweep), `POST`/`DELETE /admin/reports/schedule`.
- **Audio proctoring + room scan** — two opt-in proctoring controls (Exam Builder → Proctoring). **Audio monitoring** listens on the candidate's microphone during the exam (Web Audio API, nothing recorded) and raises a flag when sustained talking/noise is detected (debounced). **Room scan** asks the candidate at check-in to slowly pan their webcam around the room; a short series of frames is captured and saved for the proctor, shown in a new "Room scan" panel on the attempt review. New `LockdownConfig.audioMonitoring`/`requireRoomScan`, `Registration.roomScanPhotos`, `audio_noise` proctor-event type.
- **Parameterized questions** — a new question type where each candidate gets **different random numbers** in the same question (anti-cheating for quantitative exams). In the Exam Builder, author variables (name, min, max, decimals) and an **answer formula** (e.g. `d/t`), using `{name}` markers in the prompt; a live preview shows a sample. At attempt start the values are **frozen per candidate** and substituted into the prompt; grading **computes the correct answer from the formula** and accepts the candidate's number within a tolerance. Formula evaluation uses a safe in-house expression parser (`shared/expr.ts`, no `eval`) supporting `+ − * / % ^`, parentheses and functions (sqrt/round/abs/min/max/…). Review and results show each candidate's specific version + computed answer.
- **Two-factor authentication (2FA / TOTP)** — optional authenticator-app login for any account (staff or student). Enable it under **Account → Security**: scan a QR (or copy the key) into Google Authenticator / Authy / 1Password / Microsoft Authenticator, confirm a 6-digit code, and save 10 one-time **backup codes**. At sign-in, after the password the app asks for the current code (or a backup code). Self-contained TOTP (RFC 6238) implemented with Node crypto — **no new dependency**; secrets never leave the server, backup codes stored bcrypt-hashed. New `server/totp.ts`, `src/components/TwoFactorSettings.tsx`, `POST /auth/2fa/{setup,enable,disable,verify}`, a short-lived pending-2FA challenge cookie.
- **PWA / installable app** — Orcalis can now be installed to a phone's home screen or desktop (Add to Home Screen / Install). Adds a web app manifest (standalone display, charcoal theme, branded 192/512 + maskable icons), Apple touch-icon/meta tags, and a **safe service worker**: it caches only static build assets and the app shell for fast loads and offline shell, and **never touches `/api`**, so exam delivery, answer sync, proctoring and submission always stay live. The worker registers on the live site only (skipped on localhost). New `public/manifest.webmanifest`, `public/sw.js`, `public/icon-192.png`, `public/icon-512.png`.
- **Fix: "Too many attempts" on first login from a shared network** — the login/signup limiter was keyed by IP (`max 20 / 15 min`), so a whole class behind one campus/office NAT shared a single bucket and legitimate first-time logins were blocked. It's now keyed by **account (email)** — distinct users from the same IP each get their own budget, and brute-forcing one account is *stronger* (capped across all IPs). The global `/api` limiter is now keyed by **session** so many students polling during an exam from one IP don't collectively trip it. ([security.ts](server/security.ts))
- **Fix: phantom "Fullscreen Exit" flag on clean submissions** — finishing a proctored exam calls `exitFullscreen()`, which previously recorded a high-severity `fullscreen_exit` flag against the candidate (a false positive that hit every proctored submission). Now guarded three ways: the client stops recording flags once a submission/termination begins ([Session.tsx](src/pages/Session.tsx)); the server drops any proctor event for an already-submitted attempt ([index.ts](server/index.ts)); and a retroactive `cleanEvents()` filter strips any `fullscreen_exit` logged at/just-before an attempt's submit time from all scoring and display, cleaning up historical records (result page, attempt review, integrity scores, results list, exports, dashboards).
- **ID verification at check-in** — when an exam enables "Photo-ID capture" (Exam Builder → Proctoring), the candidate submits a photo of a physical photo-ID during system check (upload a file or snap it with the webcam). On the attempt-review page the proctor sees the **check-in selfie and the photo-ID side by side** and can **confirm "ID matches candidate"** (recorded with the verifier's name, audit-logged). New `Registration.idDocumentPhoto/idVerified` fields + `POST /admin/registrations/:id/verify-id`.
- **SMS / WhatsApp reminders** — the 24-hour and 1-hour exam reminders can also go out by text. **Provider-agnostic**, mock by default (logged only) so it runs with no config; set `SMS_PROVIDER=twilio` + `TWILIO_ACCOUNT_SID/AUTH_TOKEN/FROM` (and `SMS_CHANNEL=whatsapp` + `TWILIO_WHATSAPP_FROM` for WhatsApp) to deliver. Toggle in Settings ("SMS / WhatsApp reminders") with live/mock status and a test-send; status also shown on System Health. New `server/sms.ts`, `OrgSettings.smsReminders`.
- **Calendar view** — month calendar for both portals. Students see their exam timetable (colour-coded by status); admins see when each exam opens. Today highlighted, click-through to exam/check-in. New `MonthCalendar` component + `/calendar` and `/admin/calendar`.
- **Bulk student CSV import** — header-aware CSV (name, email, +class/gender/age/phone) → creates candidates, dedupes, emails temporary passwords, audit-logged. "Import CSV" modal on Students with an added/skipped summary.
- **Question Pools per section** — each exam section can serve a random subset ("draw N of M") per attempt; configured with a per-section control in the Exam Builder.
- **Study materials / Resources** — attach labelled links to an exam (revision notes, syllabus, past papers); shown to candidates on the exam page's Resources tab.
- **Item analysis** — per-question correct-rate, difficulty band and discrimination index (top vs bottom third) from real submissions. `/admin/exams/:id/analysis`, reachable from Results; searchable/filterable/exportable.
- **Notifications / inbox center** — unified feed derived from live data (results, awaiting-grading, exam reminders, announcements for students; grading queue, recent submissions, announcements for staff). `/inbox` and `/admin/inbox`, unread badges, mark-all-read.
- **Security/CSP fix** — explicit Content-Security-Policy via helmet + `public/.htaccess` (allows the eval the Monaco editor/WebAssembly need; locks down object-src/base-uri).
- **Pick from question bank** — reuse existing questions across exams. A searchable/type-filtered picker in the Exam Builder ("Pick from question bank") deep-clones selected bank questions into the current exam (`POST /exams/:id/questions/clone`).
- **Five new question types** — **Matching** (left→right, shuffled options), **Ordering/Sequence** (reorder into the correct order), **Cloze / fill-in-the-blank** (`___` markers, per-blank accepted answers, proportional grading), **Image Hotspot** (drag to mark correct regions; a candidate click inside any scores), and **File Upload** (≤5 MB, manually graded). Full pipeline: builder editors, candidate runner, auto-grading (with partial credit), review display and publish validation.
- **Topics/tags + exam blueprint** — tag questions by topic (in a question's Advanced settings); auto-assemble each attempt by blueprint ("draw N from Topic A, M from Topic B"). Tags also filter the bank picker.
- **Math / LaTeX rendering** — KaTeX renders `$…$` / `$$…$$` in prompts and options across the candidate runner, results and the builder preview (plus the toolbar's markdown: bold/italic/code/underline/images/bullets). New `MathText` renderer.
- **Duplicate exam as template + rubric library** — one-click duplicate an exam (clones the exam and all its questions as a draft, from the Library card/row). Reusable rubric library: save a question's rubric once (named) and apply it to any essay/code/file-upload question; stored on org settings.
- **File-upload download for graders** — the attempt-review page now offers a download link for uploaded answer files (data-URL passthrough), so graders can open submissions, not just see the filename.
- **Grade scaling + boundaries** — per-exam score curve (add points / multiply factor, capped) with the raw score preserved; letter-grade boundaries (A/B/C… by min %, defaults provided) shown on student and admin results; **bulk result release** per exam and **scheduled release** (results held from students until a set time); "apply scheme to existing results" recompute. New `shared/grades.ts` (`applyCurve`/`letterFor`).
- **Anonymous / double-blind grading** — per-exam flag that hides candidate identity (name, email, ID, webcam frames) from graders in the grading queue and attempt review until results are released. Lightweight **second-marker reconciliation**: record an independent second score; a ≥10-point gap is flagged for reconciliation.
- **Regrade / appeals workflow** — students can request a review of a released result (with a reason); requests are routed to a new admin **Regrade requests** queue where a grader responds, optionally adjusts the score (re-issuing a certificate on pass), and resolves or declines. New `regradeRequests` collection.
- **LMS gradebook export** — export results from the admin Results page as **Canvas**- or **Moodle**-style gradebook CSV (pivoted student × exam, latest attempt wins).
- **Exam reliability + distractor analysis** — the item-analysis page now shows **Cronbach's α** (internal-consistency reliability, with an interpretation band) and a per-option **distractor breakdown** (pick rate per option, correct one highlighted) for choice questions.
- **Cohort & term-over-term comparison + at-risk flagging** — the Analytics page compares classes (avg score) and terms (academic years, else last 6 months), and flags **at-risk students** (low average, repeated fails, or declining scores) with click-through to their record.
- **Scheduled admin email digests** — daily/weekly summary (submissions, in-progress, awaiting grading, open appeals) emailed to all admins, configurable in Settings, with a "send now" action. Server-side scheduler tick reusing the existing mailer.
- **Per-student PDF progress report** — a printable progress report per student (profile, exam history with grades, attendance roll-up, certificates), reached from the student record; "Print / Save as PDF" produces the document.
- **Live proctor intervention** — from the Live Monitor, open a session drawer to **message** the candidate (banner appears in their exam), **pause** the attempt (their timer freezes and a lock overlay appears; paused time is returned on resume), or **terminate** (force-submit) it. The drawer shows a **recording timeline** — the webcam snapshot strip with flagged moments highlighted — plus the flagged-event list, live progress and integrity. The candidate's exam polls a lightweight control channel, so pause/resume, messages and termination apply within seconds.
- **Excel and Word import** — the question import (Exam Builder) and the bulk student import now accept **.xlsx (Excel)** and **.docx (Word)** files in addition to CSV. Excel reads the first sheet; Word reads the first table; both use a header row mapped to the same columns as the CSV. The heavy parsers (SheetJS, mammoth) are lazy-loaded so they stay out of the main bundle.
- **Per-student subject trend infographic** — on the admin Student Record and in the per-student PDF report: an overall score-over-time chart (points coloured by subject, with a trend line), per-subject tiles (average, best, a sparkline and an Improving/Steady/Declining arrow from a least-squares slope), and a statistical summary (strongest/weakest subject, overall direction, subjects flagged for attention).
- **AI difficulty checker** — in the Exam Builder, a "Suggest difficulty (AI)" button next to the difficulty field reads the question (prompt, options, answer key, type, tags) and returns an easy/medium/hard estimate with a confidence and a one-line rationale, plus a one-click Apply. Works with **any OpenAI-compatible provider** (Groq, Google Gemini, OpenRouter, OpenAI — set `AI_BASE_URL`+`AI_API_KEY`+`AI_MODEL`) or **Anthropic/Claude** (`ANTHROPIC_API_KEY`, model via `ORCALIS_AI_MODEL`, default `claude-opus-4-8`). Hidden when AI isn't configured; provider libs are lazy-loaded so the server runs fine without it.
- **AI student trend narrative** — on the admin Student Record, a "Generate AI summary" turns the subject-trend data into a short natural-language paragraph for the teacher (same provider config as above).
- **Per-question explanations** — each question can carry an "Explanation (the why)" shown to the candidate on their released result and to graders during review.
- **Accommodations (extra exam time)** — set per-student extra minutes on the Student Record; added to every exam's deadline for that student (accessibility / extended time), composing with scheduled windows and proctor pauses.
- **Audit-log retention** — an org setting to auto-purge admin audit-log entries older than N days (0 = keep forever), enforced by the recurring retention sweep.
- **Certificate QR + printable certificate** — a public, branded, print-ready certificate at `/certificate/:certNumber` with a verification **QR code** (links to the public verify page); "View / Print" from the student Certificates page produces a Save-as-PDF certificate.
- **Cumulative GPA** — a 4.0-scale cumulative GPA (from exam scores) shown on the admin Student Record and the per-student PDF report.
- **Per-section time limits** — each exam section can carry a suggested time (minutes), set in the Exam Builder and shown to candidates in the section header during the exam (advisory).
- **Answer similarity / plagiarism flagging** — per exam, a similarity scan compares written answers (essay/short/code) across submissions and flags candidate pairs above a character-similarity threshold, with click-through to each attempt. Reached from the Results page ("Similarity").

**Backlog ideas (not yet built):** AI question generation, AI-assisted essay grading, full independent blind double-marking.

## v1.3.0 — UI polishing

A platform-wide design pass to make the product feel human-crafted, not AI-generated. Standing design charter: real data everywhere, enterprise density, role-aware, actionable widgets; **pumpkin #FD802E + charcoal #233D4C**, flat (no gradients), theme-aware (dark + light).

- **Typography system (redesign)** — replaced the placeholder Poppins + generic Trebuchet MS with **Space Grotesk** (display/headings) + **DM Sans** (body/UI), free Google Fonts with `display=swap`. Headings tuned to 600 weight with tight tracking; body 14px/1.5. This single change reskins every page. Grounded in the UI/UX Pro Max skill's pairings ("Tech Startup": Space Grotesk + DM Sans).
- **Foundation craft pass** — **tabular numerals** on all tables and stat figures (`font-variant-numeric: tabular-nums`) so numbers align in columns; a **distinctive Space Grotesk numeric identity** for big KPI/score figures across both dashboards, results, analytics, the exam library and attempt review; a global **`:focus-visible` ring** for keyboard accessibility; a **`prefers-reduced-motion`** block; and a pumpkin **brand keyline** + optional eyebrow on the shared `PageHeader` (used on every page).
- **Login** — password show/hide toggle and an announced (`role="alert"`) error, plus the new type system.
- **Exam runner** — the candidate timer reads in the Space Grotesk face with a low-time pulse cue.

- **Login redesign** + copy.
- **Student Dashboard, Admin Dashboard** rebuilt as **bento grids** (varied-size tiles) with right-rail panels; left-aligned at a moderate max width (not centered).
- **Exam Builder** rebuilt into a master–detail authoring tool (structure tree + question editor + Settings/Proctoring tabs), with difficulty, rich-text toolbar, autosave.
- **Exams page** rebuilt as list + detail with a Readiness Center, countdown, and tabs.
- **Exams Library** (merged exams + question bank with filters).
- **Enterprise `DataTable`** (search, sortable columns, filters, pagination, bulk actions, CSV export) rolled across Results, Grading, Attendance, Violations, Certificates, Audit Logs, Students.
- **Charcoal greeting header** across the entire student portal (name, date, theme toggle, apps launcher, bell).
- **Truth & trust fixes** — removed fabricated/ misleading widgets (fake histogram, hardcoded "operational", blanket "Verified"); silent write failures now roll back + notify.
- **One brand, one theme** — removed all gradients and stray indigo/purple; fixed light-mode chips; unified components.
- **Form polish** — inline validation + autosave (Settings, Notifications, Team, Organization); last-admin guard; spinner-glyph fix.
- **Card polish** — brighter, better-defined card surfaces and elevation in both themes.
- **Idle auto-logout** (10 min) and other microinteractions.
